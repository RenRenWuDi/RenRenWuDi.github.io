机密价值与匿名花费：ZK隐私机制

发表于2026-07-18|更新于2026-07-18|隐私DeFi
|总字数:2k|阅读时长:7分钟|浏览量:

Pedersen承诺+Bulletproofs+Groth16：隐藏金额的数学原理

1. 隐私的两个维度

tacit的隐私不是简单的”加密”。它区分了两个维度：

机密价值（Confidential Value）：隐藏转账金额，但证明金额正确且非负

匿名花费（Anonymous Spend）：断开输入UTXO和输出UTXO之间的关联

两个维度用不同的密码学工具实现：机密价值用Pedersen承诺+Bulletproofs，匿名花费用Groth16+Poseidon-Merkle+nullifiers。

2. Pedersen承诺：隐藏金额的核心

Pedersen承诺是一个经典的密码学工具，核心思想是：你可以承诺一个值而不泄露它，之后打开承诺时别人能验证你承诺的确实是这个值。

Pedersen承诺的数学：

选两个生成元 G, H（椭圆曲线上的点）
承诺 C = vG + rH
其中 v = 金额，r = 盲因子（随机数）

C 是椭圆曲线上的一个点，从C无法反推v和r（基于离散对数难题）
但知道v和r的人可以验证C确实承诺了v。

在tacit中，每笔交易的输出是一个Pedersen承诺而非明文金额。外部观察者只看到一个椭圆曲线点，不知道金额是多少。但交易双方知道金额和盲因子，可以验证正确性。

3. Bulletproofs：证明金额非负

Pedersen承诺隐藏了金额，但有个问题：怎么证明金额是正数？如果有人承诺一个负数金额，就可以凭空创造价值。

Bulletproofs解决了这个问题。它是一种零知识证明，可以证明一个承诺的值在[0, 2^n]范围内，而不泄露具体值。

Bulletproofs的关键特性：

证明大小：O(log n)，非常紧凑
不需要可信设置（trusted setup）
支持聚合证明：多个范围证明可以合并成一个，大幅节省空间

tacit使用聚合Bulletproofs，一个交易里多个输出的范围证明打包成一个，减少链上数据量。

contract.sol

守恒验证的关键：所有输入承诺的和等于所有输出承诺的和。因为盲因子也满足守恒（sum(r_in) = sum(r_out)），所以椭圆曲线点的和也守恒。这证明了”金额没有凭空创造或销毁”。

4. Mimblewimble内核签名

tacit还借鉴了Mimblewimble的设计——内核签名（Kernel Signature）。

在Mimblewimble中，一笔交易不需要传统的地址和签名，而是通过”内核”来证明交易的有效性。内核包含：

交易偏移量（kernel excess）：输入盲因子之和与输出盲因子之差的余项

内核签名：用余项作为私钥对交易进行签名

索引器验证内核签名，确认交易双方都同意这笔交易（因为只有知道盲因子的人才能生成有效签名）。这比传统的脚本签名更隐私——不暴露任何地址信息。

5. 匿名花费：混币池机制

机密价值隐藏了金额，但UTXO之间的关联仍然可见——你能看到A的UTXO被花费，然后产生了一个新UTXO给B。匿名花费要断开这个关联。

tacit的匿名花费是一个固定面额混币池（Mixer Pool）：

混币流程：

  1. 存入：把机密UTXO存入混币池，获得一个混币叶子（mixer leaf）
  2. 等待：池子里有足够多的存款后，进入提取阶段
  3. 提取：用零知识证明证明”我拥有某个混币叶子的密钥”，提取到新地址
  4. 断联：链上只有存款和取款记录，无法连接存款地址和取款地址

这套机制用到了三个密码学组件：

组件一：Groth16零知识证明

Groth16是最紧凑的zk-SNARK证明系统之一。tacit用它证明”我知道某个混币叶子的密钥”，而不泄露是哪个叶子。

证明大小：约200字节（非常小）

验证时间：几毫秒

需要可信设置：tacit已完成trusted setup ceremony，artifacts已公开发布

组件二：Poseidon-Merkle树

混币池的叶子组成一棵Merkle树，但不是传统的哈希树——而是用Poseidon哈希函数。

Poseidon是一种专门为ZK电路优化的哈希函数：

为什么不用Keccak/SHA256？

传统哈希函数在ZK电路里的约束数量很大（每个哈希运算需要数千个约束）
Poseidon在ZK电路里只需要几百个约束——快10倍以上

tacit用Poseidon-Merkle树来组织混币叶子，Groth16证明里只需要验证Merkle路径，用Poseidon大幅减少证明生成时间。

组件三：Nullifiers（废止标记）

混币池最大的风险是双重提取：同一个人多次提取同一个叶子。Nullifiers解决这个问题。

每次提取时，用户必须发布一个nullifier值——它是叶子密钥的哈希。索引器检查：

Nullifier机制：

nullifier = Poseidon(leaf_secret, mixer_id)

提取时在链上发布nullifier
索引器维护已使用nullifier集合
如果nullifier已存在 → 拒绝提取

关键：从nullifier无法反推是哪个叶子（单向哈希），但同一叶子的nullifier是唯一的。
