重入攻击
The DAO事件的元凶
系列六「智能合约安全攻防」第1篇
💡 你将学到：重入攻击原理、The DAO事件、CEI模式、ReentrancyGuard
⏱ 阅读时间：约13分钟

  1. The DAO事件回顾
    2016年6月17日，区块链史上最大的安全事件之一发生了。黑客利用以太坊上一个叫”DAO”的智能合约漏洞，偷走了360万ETH——当时价值约6000万美元。ETH价格随后暴跌，整个以太坊社区被迫硬分叉。
    这次攻击用的就是重入攻击（Reentrancy Attack）。它揭示了智能合约安全的一个根本问题：合约在执行过程中，可能被恶意调用，形成循环套娃。
    The DAO事件时间线：

• 2016年5月：DAO通过ICO融资1.5亿美元
• 2016年6月17日：黑客发动重入攻击
• 2016年7月：以太坊硬分叉，ETH分裂为ETH和ETC
• 这是区块链史上第一次大规模安全事件
2. 重入攻击原理
重入攻击的核心是：一个合约在收到ETH转账后，如果调用了一个恶意合约，恶意合约会利用回调函数再次调用原合约，形成无限循环。
想象一下：你去ATM取钱，ATM先把钱给你，但还没更新你的账户余额。这时候你快速再取一次，ATM发现余额还没变，又给你钱——直到账户余额归零。
重入攻击的三个条件：

  1. 合约状态更新发生在转账之后
  2. 向外部合约转账ETH
  3. 外部合约可以被攻击者控制
  4. 经典重入攻击代码
    VulnerableBank.sol
    // 有漏洞的银行合约
    contract VulnerableBank {
    mapping(address => uint256) public balances;
    function deposit() external payable {
    balances[msg.sender] += msg.value;
    }
    function withdraw() external {
    uint256 amount = balances[msg.sender];
    require(amount > 0, “No balance”);
    (bool success, ) = msg.sender.call{value: amount}(“”);
    require(success, “Transfer failed”);
    balances[msg.sender] = 0; // 状态更新在转账之后——漏洞！
    }
    }

攻击合约
Attacker.sol
// 攻击合约
contract Attacker {
VulnerableBank public bank;
constructor(address _bank) {
bank = VulnerableBank(_bank);
}
function attack() external payable {
bank.deposit{value: 1 ether}();
bank.withdraw();
}
receive() external payable {
if (address(bank).balance >= 1 ether) {
bank.withdraw(); // 再次调用withdraw()，形成循环
}
}
}

攻击流程图
攻击流程：
步骤1：Attacker调用bank.withdraw()，Bank检查余额并转账
步骤2：ETH转入Attacker，Attacker收到ETH触发receive()
步骤3：receive()再次调用bank.withdraw()，Bank再次转账
步骤4：循环重复，直到Bank余额耗尽
步骤5：balances[Attacker]=0最终执行，但Bank已无钱可取
4. CEI模式（Checks-Effects-Interactions）
CEI是最简单、最有效的防止重入攻击的方法。原则是：先检查条件，再更新状态，最后才与外部合约交互。
SafeBank.sol
// 修复后的Bank合约——CEI模式
contract SafeBank {
mapping(address => uint256) public balances;
function withdraw() external {
// 第一步：检查（Check）
uint256 amount = balances[msg.sender];
require(amount > 0, “No balance”);
// 第二步：更新状态（Effect）
balances[msg.sender] = 0;
// 第三步：与外部合约交互（Interaction）
(bool success, ) = msg.sender.call{value: amount}(“”);
require(success, “Transfer failed”);
}
}

  1. ReentrancyGuard（互斥锁）
    OpenZeppelin提供了ReentrancyGuard，用一个简单的修饰器来防止重入。
    SecureBank.sol
    // SPDX-License-Identifier: MIT
    pragma solidity ^0.8.20;
    import “@openzeppelin/contracts/security/ReentrancyGuard.sol”;
    contract SecureBank is ReentrancyGuard {
    mapping(address => uint256) public balances;
    function withdraw() external nonReentrant {
    uint256 amount = balances[msg.sender];
    require(amount > 0);
    balances[msg.sender] = 0;
    (bool success, ) = msg.sender.call{value: amount}(“”);
    require(success);
    }
    }

  2. 不同转账方式的区别
    Solidity中有三种ETH转账方式，安全性各不相同。
    三种转账方式对比：

• transfer()：最多2300 gas，失败会revert。安全但限制了合约接收ETH的能力。
• send()：最多2300 gas，失败返回false。需要检查返回值。
• call()：转发所有gas，失败返回false。最灵活但最危险，推荐配合ReentrancyGuard使用。
7. 真实重入攻击案例
The DAO（2016）：损失360万ETH，ETH硬分叉
Cream Finance（2021）：损失1.3亿美元，重入攻击
Siren Protocol（2022）：NFT借贷平台重入漏洞
8. 总结

① The DAO事件是历史上最大的重入攻击
② 重入攻击核心：状态更新在外部调用之后
③ CEI模式：先检查→再更新状态→最后交互
④ ReentrancyGuard用互斥锁彻底防止重入
⑤ call()最灵活但最危险，要配合Guard使用
⑥ 每笔外部调用都要考虑重入风险
