权限控制
onlyOwner、访问控制列表、角色权限管理
系列二「Solidity智能合约开发」第5篇
💡 你将学到：Modifiers、Ownable模式、RBAC角色权限、OpenZeppelin AccessControl
⏱ 阅读时间：约15分钟

  1. 为什么需要权限控制？
    智能合约一旦部署就无法修改。如果有敏感函数没有权限控制，任何人都可以调用，可能导致资金被盗或合约逻辑被破坏。
    权限控制的必要性：

• 管理员函数（暂停、升级、铸造）只有特定人能调用
• 防止误操作导致资金损失
• 满足合规要求（如KYC、白名单）
• 实现代币的mint权限控制
常见需要权限的函数
mint/burn：铸造和销毁代币
pause/unpause：暂停合约
upgrade：升级合约逻辑
withdraw：提取合约中的资金
setFee/setConfig：修改配置参数
2. Modifier（修饰器）
Modifier是Solidity中实现权限控制的核心语法。它可以复用检查逻辑，让代码更简洁。
Modifier的特点：

• 在函数执行前运行检查
• _; 代表执行被修饰的函数体
• 可以组合多个modifier
• 比在函数内写require更优雅
基本用法
ModifierExample.sol
contract ModifierExample {
address public owner;

constructor() {
owner = msg.sender;
}

// 定义modifier
modifier onlyOwner() {
require(msg.sender == owner, “Not owner”);
_; // 执行函数体
}

// 使用modifier
function sensitiveAction() public onlyOwner {
// 只有owner能执行这里
}

function publicAction() public {
// 任何人都能执行
}
}

带参数的Modifier
RoleModifier.sol
contract RoleModifier {
mapping(address => bool) public admins;

modifier onlyAdmin(address _admin) {
require(admins[_admin], “Not admin”);
_;
}

function restrictedAction() public onlyAdmin(msg.sender) {
// …
}
}

组合Modifier
ComposedModifier.sol
contract ComposedModifier {
address public owner;
bool public paused;

modifier onlyOwner() {
require(msg.sender == owner);
_;
}

modifier whenNotPaused() {
require(!paused, “Contract is paused”);
_;
}

// 组合多个modifier
function sensitiveAction() public onlyOwner whenNotPaused {
// 只有owner能执行，且合约未暂停
}
}

  1. Ownable模式
    Ownable是最简单的权限模式：只有一个owner，owner可以转移权限。
    Ownable模式：

• 单一owner，所有权限集中
• 适合简单场景
• OpenZeppelin提供了标准实现
手动实现Ownable
Ownable.sol
contract Ownable {
address public owner;

event OwnershipTransferred(address indexed previousOwner, address indexed newOwner);

constructor() {
owner = msg.sender;
}

modifier onlyOwner() {
require(msg.sender == owner, “Ownable: caller is not the owner”);
_;
}

function transferOwnership(address newOwner) public onlyOwner {
require(newOwner != address(0), “Ownable: new owner is the zero address”);
address oldOwner = owner;
owner = newOwner;
emit OwnershipTransferred(oldOwner, newOwner);
}
}

使用Ownable
MyToken.sol
contract MyToken is Ownable {
uint256 public totalSupply;
mapping(address => uint256) public balanceOf;

// 只有owner能铸造代币
function mint(address to, uint256 amount) public onlyOwner {
totalSupply += amount;
balanceOf[to] += amount;
}

// 任何人都能转账
function transfer(address to, uint256 amount) public {
// 转账逻辑…
}
}

  1. RBAC：基于角色的访问控制
    Ownable只适合单一owner场景。复杂应用需要多角色，如admin、minter、pauser。
    RBAC（Role-Based Access Control）：

• 将权限分配给角色
• 用户可以被分配多个角色
• 角色可以有多个成员
• 更灵活，适合复杂业务逻辑
手动实现RBAC
SimpleRBAC.sol
contract SimpleRBAC {
// 角色定义
bytes32 public constant ADMIN_ROLE = keccak256(“ADMIN_ROLE”);
bytes32 public constant MINTER_ROLE = keccak256(“MINTER_ROLE”);
bytes32 public constant PAUSER_ROLE = keccak256(“PAUSER_ROLE”);

// 角色成员映射
mapping(bytes32 => mapping(address => bool)) public roles;

modifier onlyRole(bytes32 role) {
require(hasRole(role, msg.sender), “AccessControl: unauthorized”);
_;
}

function hasRole(bytes32 role, address account) public view returns (bool) {
return roles[role][account];
}

function grantRole(bytes32 role, address account) public onlyRole(ADMIN_ROLE) {
_grantRole(role, account);
}

function _grantRole(bytes32 role, address account) internal {
roles[role][account] = true;
emit RoleGranted(role, account, msg.sender);
}

event RoleGranted(bytes32 indexed role, address indexed account, address indexed sender);
}

使用RBAC
MyTokenWithRBAC.sol
contract MyTokenWithRBAC is SimpleRBAC {
uint256 public totalSupply;
mapping(address => uint256) public balanceOf;
bool public paused;

constructor() {
_grantRole(ADMIN_ROLE, msg.sender);
_grantRole(MINTER_ROLE, msg.sender);
}

function mint(address to, uint256 amount) public onlyRole(MINTER_ROLE) {
totalSupply += amount;
balanceOf[to] += amount;
}

function pause() public onlyRole(PAUSER_ROLE) {
paused = true;
}

function transfer(address to, uint256 amount) public {
require(!paused, “Paused”);
// 转账逻辑…
}
}

  1. OpenZeppelin AccessControl
    OpenZeppelin是最广泛使用的智能合约库，提供了标准化的AccessControl实现。
    OpenZeppelin AccessControl优势：

• 经过审计，安全可靠
• 完整的角色管理
• 支持枚举类型查询角色成员
• 与其他OZ合约无缝集成
使用OZ AccessControl
MyTokenOZ.sol
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

import “@openzeppelin/contracts/access/AccessControl.sol”;

contract MyToken is AccessControl {
bytes32 public constant MINTER_ROLE = keccak256(“MINTER_ROLE”);
bytes32 public constant PAUSER_ROLE = keccak256(“PAUSER_ROLE”);

constructor() {
_grantRole(DEFAULT_ADMIN_ROLE, msg.sender);
_grantRole(MINTER_ROLE, msg.sender);
}

function mint(address to, uint256 amount) public onlyRole(MINTER_ROLE) {
_mint(to, amount);
}

function grantMinterRole(address account) public onlyRole(DEFAULT_ADMIN_ROLE) {
grantRole(MINTER_ROLE, account);
}

function revokeMinterRole(address account) public onlyRole(DEFAULT_ADMIN_ROLE) {
revokeRole(MINTER_ROLE, account);
}
}

  1. 安全注意事项
    权限控制常见漏洞：

• 忘记加onlyOwner
• owner地址设为0x0无法恢复
• transferOwnership后可立即转走权限
• 合约创建者权限未转移
最佳实践
敏感函数必须添加权限检查
使用经过审计的库，如OpenZeppelin
考虑权限时间锁：TimelockController
实现紧急暂停机制
权限变更应发出事件，便于监控
测试覆盖所有权限分支
时间锁（Timelock）
TimelockController.sol
// 重要操作先发起，经过延迟后才能执行
contract TimelockController {
uint256 public constant MIN_DELAY = 2 days;
mapping(bytes32 => uint256) public timestamps;

function schedule(bytes32 actionHash) public onlyOwner {
timestamps[actionHash] = block.timestamp + MIN_DELAY;
}

function execute(bytes32 actionHash) public onlyOwner {
require(block.timestamp >= timestamps[actionHash], “Too early”);
// 执行操作…
}
}

  1. 总结

① Modifier是实现权限控制的核心语法
② Ownable适合单一owner的简单场景
③ RBAC支持多角色，适合复杂业务
④ OpenZeppelin AccessControl是行业标准
⑤ 重要操作考虑多签和时间锁
⑥ 使用经过审计的库比自己实现更安全
