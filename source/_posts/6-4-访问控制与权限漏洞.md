访问控制与权限漏洞
谁可以调用这个函数
系列六「智能合约安全攻防」第4篇
💡 你将学到：权限绕过、初始化漏洞、constructor陷阱、AccessControl最佳实践
⏱ 阅读时间：约12分钟
1. 访问控制为什么重要？
区块链上，所有数据都是公开的，任何人都可以调用任何公开函数。如果一个本应只有管理员才能调用的函数没有做权限控制，那么任何人都可以用它。
访问控制漏洞往往不是代码逻辑本身有问题，而是"谁可以调用这个函数"这个问题没有被正确回答。
访问控制漏洞的严重性：

• 2022年，约37%的DeFi攻击源于访问控制漏洞
• 很多漏洞不是因为逻辑错误，而是权限遗漏
• 越隐蔽的漏洞越危险
• 访问控制是合约安全的第一道防线
2. 常见的访问控制模式
Ownable（单一管理员）
最简单的模式：合约有一个owner，onlyOwner修饰器控制权限。
Ownable.sol
contract Ownable {
address public owner;
modifier onlyOwner() {
require(msg.sender == owner, "Not owner");
_;
}
function renounceOwnership() public onlyOwner {
owner = address(0);
}
}

AccessControl（多角色）
更灵活的RBAC模式，支持多角色和权限组。
AccessControl.sol
import "@openzeppelin/contracts/access/AccessControl.sol";
contract MyContract is AccessControl {
bytes32 public constant ADMIN_ROLE = keccak256("ADMIN_ROLE");
bytes32 public constant MINTER_ROLE = keccak256("MINTER_ROLE");
function grantAdmin(address account) external {
grantRole(ADMIN_ROLE, account);
}
function mint(address to, uint256 amount) external {
require(hasRole(MINTER_ROLE, msg.sender));
// mint logic...
}
}

3. 权限绕过漏洞
最常见的漏洞是：忘记给某个关键函数加上权限修饰器。
BadVault.sol
// 漏洞代码——withdrawFunds忘记加onlyOwner
contract BadVault is Ownable {
mapping(address => uint256) public balances;
function withdraw(uint256 amount) external {
require(balances[msg.sender] >= amount);
balances[msg.sender] -= amount;
payable(msg.sender).transfer(amount);  // 任何人都能调用！
}
}

修复很简单，但遗漏的后果很严重——攻击者可以偷走所有人的存款。
4. Constructor初始化漏洞
Proxy合约（升级合约）中，constructor只在新合约部署时执行一次。如果用Proxy模式部署多个实例，旧合约的owner可能还是部署者地址，但新合约的owner却设置了。
Proxy模式中的initialize()陷阱：

• Proxy合约本身没有构造函数
• 初始化逻辑放在initialize()函数中
• 如果initialize()没有权限保护
• 任何人都可以调用它，将自己设为owner
• 2022年很多合约因此被盗
BadProxy.sol
// 漏洞代码——没有 initializer 修饰器
contract BadProxy {
address public owner;
uint256 public value;
function initialize() public {
owner = msg.sender;  // 任何人都能调用！
}
}

正确做法
GoodProxy.sol
import "@openzeppelin/contracts-upgradeable/proxy/utils/Initializable.sol";
contract GoodProxy is Initializable {
address public owner;
uint256 public value;
function initialize() public initializer {
owner = msg.sender;  // 只在部署时调用一次
}
}

5. tx.origin陷阱
tx.origin返回交易的原始发送者，而不是直接的调用者。在某些场景下，用tx.origin做权限检查是危险的。
TxOrigin.sol
// 危险的代码
function adminTransfer(address to, uint256 amount) external {
require(tx.origin == contractOwner);  // 危险！
// ...
}
// 如果用户通过一个恶意合约调用这个函数
// 恶意合约调用adminTransfer时，tx.origin仍然是用户
// 攻击合约可以绕过tx.origin检查

正确做法：用msg.sender替代tx.origin。tx.origin几乎没有合法使用场景。
6. 真实权限攻击案例
Opyn（2021）：合约升级函数没有权限保护，损失370万美元
Build finance（2021）：Proxy的initialize()没有保护，攻击者获得mint权限
XCarnival（2022）：breeze合约权限绕过漏洞，损失380万美元
Build Finance事件
2022年5月，Build Finance的治理合约在升级时，新合约的initialize()函数没有加权限保护。攻击者调用initialize()获得ADMIN权限后，恶意铸造了大量BUILD代币并在DEX砸盘。这枚治理代币变得一文不值。
7. 访问控制检查清单
每个关键函数都加了权限修饰器？
initialize()函数用了initializer修饰器？
使用了tx.origin吗？考虑用msg.sender替代
owner权限是否可以转移或撤销？
使用了OpenZeppelin的标准库而不是自己写？
8. 总结

①  访问控制是合约安全的第一道防线
②  权限绕过最常见：忘记给函数加修饰器
③  Proxy模式的initialize()必须用initializer保护
④  tx.origin几乎永远不应该用于权限检查
⑤  OpenZeppelin AccessControl比Ownable更安全灵活
⑥  每个关键函数上线前都要审查访问控制
