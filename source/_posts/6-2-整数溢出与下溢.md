整数溢出与下溢
Solidity的数值陷阱
系列六「智能合约安全攻防」第2篇
💡 你将学到：溢出原理、SafeMath、0.8内置检查、unckecked陷阱
⏱ 阅读时间：约12分钟
1. 什么是整数溢出？
计算机中的整数有固定大小。以太坊使用256位整数（uint256），这意味着它的取值范围是0到2^256-1。超过这个范围，就会发生溢出——数值突然变成一个很小的数，或者归零。
打个比方：汽车的里程表跑到999999公里后，再加1公里就会归零显示0公里。整数溢出就是这种"归零"现象。
整数溢出分两种：

• 上溢（Overflow）：最大值+1=0或最小值
• 下溢（Underflow）：最小值-1=最大值

Solidity 0.8之前：不报错，静默归零
Solidity 0.8之后：自动revert
2. 溢出攻击示例
用一个简单的代币合约来演示溢出攻击。
OverflowToken.sol
// SPDX-License-Identifier: MIT
pragma solidity ^0.7.6;  // 旧版本，有溢出漏洞
contract OverflowToken {
mapping(address => uint256) balances;
function mint(address to, uint256 amount) external {
balances[to] += amount;  // 可能溢出
}
function transfer(address to, uint256 amount) external {
require(balances[msg.sender] >= amount);
balances[msg.sender] -= amount;  // 可能下溢
balances[to] += amount;
}
}

3. SafeMath：溢出保护的旧方案
Solidity 0.8之前，没有内置的溢出检查。OpenZeppelin设计了SafeMath库，通过require在计算前检查边界。
SafeMath.sol
// SafeMath库
library SafeMath {
function add(uint256 a, uint256 b) internal pure returns (uint256) {
uint256 c = a + b;
require(c >= a, "SafeMath: addition overflow");
return c;
}
function sub(uint256 a, uint256 b) internal pure returns (uint256) {
require(b <= a, "SafeMath: subtraction overflow");
return a - b;
}
}

使用时，用.safeAdd()替代+，用.safeSub()替代-。
4. Solidity 0.8内置溢出检查
从Solidity 0.8开始，编译器会自动在算术运算前后插入溢出检查。如果发生溢出，交易直接revert。
Solidity 0.8的溢出行为：

• uint256 max + 1 → revert
• int256 min - 1 → revert
• 不需要SafeMath
• 但降低了gas效率（0.8.0-0.8.19有额外gas消耗）
ModernToken.sol
// Solidity 0.8+，不需要SafeMath
contract ModernToken {
mapping(address => uint256) public balances;
function transfer(address to, uint256 amount) external {
require(balances[msg.sender] >= amount);
balances[msg.sender] -= amount;  // 编译器自动检查，下溢会revert
balances[to] += amount;
}
}

5. unchecked陷阱
Solidity 0.8引入了unchecked关键字，可以手动绕过溢出检查。但这非常危险，用错就会引入漏洞。
UncheckedDemo.sol
// unchecked示例——危险！
function dangerous(uint256 i) public pure returns (uint256) {
unchecked {
return i + 1;  // 不检查溢出
}
}
// 正确用法：已知不会溢出的情况
function counter() internal pure returns (uint256) {
uint256 c = 0;
unchecked { c++; }  // 从0开始，永不溢出
return c;
}

6. 真实溢出攻击案例
PoWH 3D（2017）：代币合约整数下溢，攻击者铸造大量代币
BatchTransfer（2018）：BEC代币溢出漏洞，一夜归零
Beauty Chain（2018）：同上，攻击者获利670万美元
BEC代币事件
2018年4月，Beauty Chain的BEC代币因为一个整数溢出漏洞，攻击者凭空铸造了天量代币，BEC价格瞬间归零。这是整数溢出在代币合约中最著名的攻击案例。
BEC.js
// BEC漏洞代码（简化）
function batchTransfer(address[] recipients, uint256 amount) external {
require(recipients.length > 0 && recipients.length <= 20);
require(amount > 0);
uint256 totalAmount = amount * recipients.length;  // 溢出！
require(balances[msg.sender] >= totalAmount);
// 后续转账...
}
// 当amount=2, recipients.length=578960446186580977117854925043439539266...
// totalAmount=0，可以通过余额检查

7. 未检查返回值
和溢出相关的另一个常见漏洞：调用外部合约后，不检查返回值。
TransferCheck.sol
// 危险的转账代码
function withdraw() external {
uint256 amount = balances[msg.sender];
balances[msg.sender] = 0;
msg.sender.transfer(amount);  // 不检查返回值！
}
// 安全的写法
function withdraw() external {
uint256 amount = balances[msg.sender];
balances[msg.sender] = 0;
(bool success, ) = msg.sender.call{value: amount}("");
require(success, "Transfer failed");
}

8. 总结

①  整数溢出/下溢是Solidity最经典的安全漏洞
②  Solidity 0.8之前需要SafeMath库手动保护
③  Solidity 0.8之后内置溢出检查，不需要SafeMath
④  unchecked关键字很危险，必须确保不会溢出才能用
⑤  外部调用后一定要检查返回值
⑥  BEC等事件提醒我们：溢出漏洞可以清零代币
