以太坊账户模型
EOA vs 合约账户
状态树与nonce机制详解
💡 你将学到：EOA与合约账户区别、nonce机制、状态存储原理
⏱ 阅读时间：约12分钟
1. 以太坊的两种账户
比特币只有UTXO，没有账户概念。但以太坊不同，它采用了账户模型。
以太坊有两种账户：
EOA：外部拥有账户（Externally Owned Account）
由私钥控制，对应真实用户
可以发起交易（转账或调用合约）
有余额（balance）和交易计数（nonce）
没有代码，不能执行逻辑
合约账户（Contract Account）
由代码控制，没有私钥
不能主动发起交易，只能被动响应
有余额、nonce、代码（code）和存储（storage）
可以执行复杂的业务逻辑
关键区别：

• EOA有私钥，人可以控制
• 合约账户有代码，逻辑控制
• 只有EOA能主动发起交易
2. 账户状态：以太坊存储了什么？
每个以太坊账户都有以下状态：
EOA状态
ethereum-account.js
{
nonce: 5,           // 交易计数（防重放攻击）
balance: 1000000000000000000,  // 余额（wei，1 ETH = 10^18 wei）
codeHash: null,     // EOA没有代码
storageRoot: null   // EOA没有存储
}

合约账户状态
ethereum-account.js
{
nonce: 1,           // 合约创建时的nonce（通常=1）
balance: 500000000000000000,   // 合约余额
codeHash: "0xabc...", // 合约代码的哈希
storageRoot: "0xdef..." // 存储数据的Merkle根
}

💡 所有账户状态都存储在"状态树"（State Trie）中，这是一棵Merkle Patricia Trie。
3. Nonce：交易顺序与防重放
Nonce是账户发出的交易数量。它有两个重要作用：
保证交易按顺序执行：交易必须按nonce递增顺序打包
防止重放攻击：同一笔交易不能被重复执行
代码示例：Nonce管理
ethereum-account.js
class Account {
constructor(address, privateKey) {
this.address = address
this.privateKey = privateKey
this.nonce = 0  // 从0开始
this.balance = 0
}

// 发送交易
sendTransaction(to, value, data = "") {
const tx = {
from: this.address,
to: to,
value: value,
data: data,
nonce: this.nonce,  // 使用当前nonce
gasPrice: 20000000000,
gasLimit: 21000
}

// 签名交易
const signedTx = this.sign(tx)

// nonce+1，为下一笔交易做准备
this.nonce++

return signedTx
}

sign(tx) {
// 使用私钥签名
return "0x" + signWithPrivateKey(tx, this.privateKey)
}
}

Nonce使用规则：

• 第一笔交易nonce=0
• 第二笔交易nonce=1
• 以此类推...

如果nonce=2的交易先被打包，nonce=3的交易必须等它完成后才能执行。
4. 状态树：以太坊的全局状态
以太坊使用Merkle Patricia Trie（MPT）存储所有账户状态。
为什么用MPT？
快速查询：O(log n)时间找到任意账户
验证方便：只需Merkle证明就能验证账户状态
增量更新：只修改变化的节点，效率高
状态树结构
ethereum-account.js
State Root（状态根）
|
[MPT Root]
/   |   \
[A]  [B]  [C]   <- 分支节点（16进制，16个子节点）
/       \
[0x1...]  [0x2...]  <- 叶子节点（账户地址 -> 账户状态）
|          |
Account1   Account2

💡 每个区块头都包含一个stateRoot，代表该区块确认后的全局状态。轻节点只需下载区块头，就能验证任意账户状态。
5. EOA vs 合约账户对比
控制方式  EOA: 私钥签名    合约: 代码逻辑
发起交易  EOA: ✅ 可以    合约: ❌ 不可以
接收转账  EOA: ✅ 可以    合约: ✅ 可以
存储数据  EOA: ❌ 不可以    合约: ✅ 可以
执行代码  EOA: ❌ 不可以    合约: ✅ 可以
创建方式  EOA: 私钥生成    合约: EOA部署
6. 实际场景：用户与合约交互
当用户使用MetaMask转账时，发生了什么？
👤  用户用EOA发起交易（nonce=5）  ↓
📝  交易包含：to、value、gas、data、签名  ↓
📡  广播到网络，矿工打包  ↓
⛏  矿工验证：签名有效、nonce正确、余额足够  ↓
✅  执行交易，更新状态树
如果to是合约地址：
EVM执行合约代码
合约可以修改自己的storage
合约可以调用其他合约
合约可以发送ETH
7. 总结

①  以太坊有EOA和合约账户两种类型
②  EOA由私钥控制，可以主动发起交易
③  合约账户由代码控制，被动响应调用
④  Nonce保证交易顺序和防重放攻击
⑤  状态树（MPT）存储所有账户状态，支持快速验证
